The key clue is this message:

```text
[src/libmpg123/frame.c:INT123_frame_outbuffer():196]
error: have external buffer of size 2824, need 4240
```

This is **not a Java buffer problem** and usually not something you can directly configure through VLCJ.

The error originates from **libmpg123**, which VLC uses internally for MP3 decoding. It indicates that the decoder encountered a frame requiring a larger output buffer than the one currently allocated. This often occurs when VLC thinks it is decoding a valid MP3 stream but the stream is actually:

* Corrupted
* Truncated
* Misidentified
* Contains garbage bytes
* Contains abrupt MPEG parameter changes (sample rate, layer, bitrate, channels)

The preceding messages strongly support this:

```text
Illegal Audio-MPEG-Header ...
not attempting to resync...

Big change (MPEG version, layer, rate).
Frankenstein stream?
```

"Frankenstein stream" is mpg123's way of saying:

> "This file appears to contain incompatible MPEG frames stitched together."

---

## First thing to check

Run ffprobe on the file:

```bash
ffprobe yourfile.mp3
```

or

```bash
ffmpeg -v error -i yourfile.mp3 -f null -
```

You'll often discover:

* Invalid frames
* Corrupt tags
* Partial downloads
* Mixed MPEG versions

---

## VLCJ does not expose this buffer

VLCJ is essentially a Java wrapper around:

```text
Java
  ↓
VLCJ
  ↓
libvlc
  ↓
ffmpeg/mpg123
```

The "external buffer" mentioned is allocated inside mpg123, not inside Java.

There is no VLCJ API like:

```java
player.setMp3BufferSize(...)
```

nor a LibVLC option that directly maps to that mpg123 buffer.

---

## If you're using audio callbacks

One exception:

If you're using VLCJ's audio callback API:

```java
AudioCallback
```

or

```java
AudioCallbackAdapter
```

and supplying your own native buffer, then the message may indicate that *your callback buffer is too small*.

For example:

```java
ByteBuffer audioBuffer =
    ByteBuffer.allocateDirect(2824);
```

when VLC suddenly requires:

```text
4240 bytes
```

In that case allocate significantly larger buffers:

```java
ByteBuffer audioBuffer =
    ByteBuffer.allocateDirect(16384);
```

or dynamically resize.

---

## If you're simply playing media

For normal playback:

```java
mediaPlayer.media().play(file);
```

there is usually nothing to tune.

Instead investigate the media itself.

## I'd specifically suspect corruption

The sequence:

```text
Illegal Audio-MPEG-Header
Big change (MPEG version, layer, rate)
Frankenstein stream
```

is almost always seen when:

1. The MP3 file is damaged.
2. An MP3 was improperly concatenated.
3. A non-MP3 file was renamed `.mp3`.
4. The file contains malformed ID3/tag data.
5. The file is only partially downloaded or partially copied.

# Step 2 — Refactor `SongMediaPlayer`

```java id="36m3go"
package com.djt.jukeanator_engine.domain.songplayer.service.utils;

import java.util.concurrent.atomic.AtomicReference;

import com.djt.jukeanator_engine.domain.songplayer.model.SongPlayerStatus;

import uk.co.caprica.vlcj.player.component.EmbeddedMediaPlayerComponent;

public final class SongMediaPlayer {

    private final EmbeddedMediaPlayerComponent mediaPlayerComponent;

    private final AtomicReference<SongPlayerStatus> status =
            new AtomicReference<>(SongPlayerStatus.STOPPED);

    public SongMediaPlayer() {

        this.mediaPlayerComponent =
                new EmbeddedMediaPlayerComponent();

        mediaPlayerComponent.mediaPlayer().events().addMediaPlayerEventListener(
                new uk.co.caprica.vlcj.player.base.MediaPlayerEventAdapter() {

                    @Override
                    public void playing(
                            uk.co.caprica.vlcj.player.base.MediaPlayer mediaPlayer) {

                        status.set(SongPlayerStatus.PLAYING);
                    }

                    @Override
                    public void paused(
                            uk.co.caprica.vlcj.player.base.MediaPlayer mediaPlayer) {

                        status.set(SongPlayerStatus.PAUSED);
                    }

                    @Override
                    public void stopped(
                            uk.co.caprica.vlcj.player.base.MediaPlayer mediaPlayer) {

                        status.set(SongPlayerStatus.STOPPED);
                    }

                    @Override
                    public void finished(
                            uk.co.caprica.vlcj.player.base.MediaPlayer mediaPlayer) {

                        status.set(SongPlayerStatus.STOPPED);
                    }
                });
    }

    public boolean playSongMedia(String songPath) {

        return mediaPlayerComponent.mediaPlayer()
                .media()
                .play(songPath);
    }

    public void pause() {

        mediaPlayerComponent.mediaPlayer().controls().pause();
    }

    public void stop() {

        mediaPlayerComponent.mediaPlayer().controls().stop();
    }

    public SongPlayerStatus getStatus() {

        return status.get();
    }

    public long getElapsedSeconds() {

        return mediaPlayerComponent.mediaPlayer()
                .status()
                .time() / 1000;
    }

    public long getTotalLengthSeconds() {

        return mediaPlayerComponent.mediaPlayer()
                .media()
                .info()
                .duration() / 1000;
    }

    public boolean isPlaying() {

        return status.get() == SongPlayerStatus.PLAYING;
    }

    public void release() {

        mediaPlayerComponent.release();
    }
}
```

# Step 7 — Process Queue

```java id="8dqgto"
private void processQueue() {

    try {

        if (mediaPlayer.isPlaying()) {
            return;
        }

        List<SongQueueEntryEntity> queue =
                songQueueRoot.getSongs();

        if (queue.isEmpty()) {
            return;
        }

        SongQueueEntryEntity nextSong =
                queue.getFirst();

        songQueueRoot.removeSongFromQueue(nextSong);

        nowPlayingSong = nextSong;

        String songPath =
                nextSong.getSong().getNaturalIdentity();

        log.info("Playing song: {}", songPath);

        mediaPlayer.playSongMedia(songPath);

    } catch (Exception e) {

        log.error("Queue processing failed", e);
    }
}
```

# Important Bug in Your Existing Queue Logic

This is dangerous:

```java id="t37m4h"
TreeSet<SongQueueEntryEntity>
```

because your `compareTo()` only compares priority:

```java id="zq4gff"
return thisPriority.compareTo(thatPriority);
```

Meaning:

* two songs with same priority are considered duplicates
* one disappears from the queue

You need:

```java id="nqsk7f"
@Override
public int compareTo(AbstractEntity obj) {

    SongQueueEntryEntity that = (SongQueueEntryEntity) obj;

    int priorityCompare =
            this.priority.compareTo(that.priority);

    if (priorityCompare != 0) {
        return priorityCompare;
    }

    return this.song.getNaturalIdentity()
            .compareTo(that.song.getNaturalIdentity());
}
```

Otherwise your queue loses songs silently.
