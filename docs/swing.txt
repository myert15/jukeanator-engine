Given the current state of the code attached, which is a commercial digital jukebox application, the following needs to be done:

1: Since the user interface will take up the entire screen and the physical keyboard will not be used, instead a bill acceptor will be connected to the computer via a keyboard encoder, and when a dollar is accepted, there will be a pulse sent to the keyboard encoder that will then emit a keystroke to the computer/application.  This keystroke character, given by the incrementCreditsKey in JukeANatorFrame will represent a dollar being inserted, thus credits need to be given.  The number of credits is  calculated by the value of the creditsPerDollar property.  So, if creditsPerDollar is 3 and an 'a' keystroke is  input, then the credits will  be incremented by 3.  When the application starts, the initial value for this numCredits property will be 0.  When 5 dollars are added to the jukebox, which would be when numCredits  = 5 x creditsPerDollar, then fiveDollarBonusCredits will be added to numCredits (in order to signify bonus credits when that much money is added to the jukebox.  Similary, when numCredits = 10 x creditsPerDollar, then tenDollarBonusCredits will be added to numCredits.

2: The whole purpose of the jukebox is to allow a user to enter money into the jukebox, get credits and then use those credits to play songs.  So, on AddSongToQueueDialog, it needs to know about the numCredits property (which should be changed to an Integer wrapper class so it is passed by reference from one method to another).  If numCredits is at at least 1, then the user can play the song at normal cost, which is 1 credit.  Similarly, if numCredits is at least priorityCost, then the user can play the song at priorityCost.  Either way, the cost will be decremented from numCredits.  The concept is the same for AddAlbumToQueueDialog, except that the normalCost is the number of songs, and the priorityCost is determined by the number of songs and the highest priority in the queue times a priority cost multiplied.  When an album is played, then numCredits should be decremented.  numCredits should NEVER be decremented below zero for either dialog.  When the user does NOT have enough, then whatever button represents a value where the user does not have enough credits in the system, then the button should be rendered like the given AMI User Interface screenshot for its add to queue behavior.  Let's call this a disabled state and the button should be disabled. It is rendered to have a red border and a second line of text that says how many credits need to be entered in order to enable the button, e.g. "ADD 2 CREDITS".  Now, if the user adds credits to the jukebox and this dialog is still visible, then the dialog needs to be notified of credits being added and dynamically enable the buttons if enough credits exist

3: Lastly, the credits label, which currently is hardcoded as: "JLabel creditsTitle = new JLabel("CREDITS: 12");" needs to be changed in order to display whatever the current value of numCredits is, as opposed to this hardcoded "12" value
